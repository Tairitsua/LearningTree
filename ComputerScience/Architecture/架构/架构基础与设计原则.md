# 架构基础与设计原则

## 架构概述

### 架构种类

#### 业务架构（`Business Architecture`, `BA`）

业务架构是对业务的结构化表达，描述组织如何运用业务关键要素实现战略目标。它通常由价值流、业务能力和业务流程等要素组成。

在规划阶段，团队可以从价值流出发，识别每一个价值流所需的关键业务能力，并进一步分析哪些能力适合借助数字技术进行重构。

#### 信息架构（`Information Architecture`, `IA`）

信息架构以结构化方式描述业务运作和管理决策所需要的信息，以及这些信息之间的关系。

业务对象是信息架构的核心。规划阶段通常会关注“产品、客户、合同、订单、员工”等关键业务对象，分析：

1. 这些对象是否已经在 `IT` 系统中被管理
2. 对象在系统之间的传递是否顺畅
3. 是否已经在数字世界中形成较完整的数字镜像

#### 应用架构（`Application Architecture`, `AA`）

应用架构识别和定义支撑业务目标所需的 `IT` 系统，以及这些系统的定位和集成关系。

规划阶段的重点通常包括：

1. 用什么联接平台构建客户和用户体验
2. 哪些 `IT` 系统承载关键业务能力
3. 不同应用之间如何集成

#### 技术架构（`Technology Architecture`, `TA`）

技术架构定义了一系列技术组件，代表企业可以从市场或内部获得的 `IT` 平台和基础设施资源。

规划阶段通常需要关注：

1. 企业应该引入哪些数字技术
2. 不同业务场景对平台和基础设施的要求

## 架构设计

架构设计的关注点通常包括但不限于：

1. 服务化
2. 异步
3. 高可用
4. 可扩展

参考链接：

1. [Design Patterns](https://design-patterns.readthedocs.io/)
2. [设计模式 GitHub 项目](https://github.com/me115/design_patterns)

设计模式的本质是一套可复用的设计思想，而不是固定写法。很多示意图和实现方式都只是典型示例。

## 设计原则

### 开闭原则

一个类或一个系统应该对扩展开放、对修改关闭。理想状态下，可以在不修改既有源码的前提下扩展行为。

### 单一职责原则（`Single Responsibility Principle`, `SRP`）

英文原句：

```txt
There should never be more than one reason for a class to change.
```

一个类最好只有一个导致它变化的原因。

## 依赖注入与控制反转

控制反转（`Inversion of Control`, `IoC`）是一种面向对象编程思想，用于降低耦合。它屏蔽依赖对象的实现和创建细节，让调用方关注“使用什么”，而不是“如何创建”。

常见实现方式有：

1. 依赖注入（`Dependency Injection`, `DI`）
2. 依赖查找（`Dependency Lookup`）

二者的区别：

1. `IoC` 更偏原理，强调对象创建控制权从代码转移到容器。
2. `DI` 更偏实现，是落地 `IoC` 的常见手段。

在 `DI` 中，构造对象时会自动注入其依赖的抽象接口实现。调用方在使用接口对象时，不需要关心对象内部依赖是如何被创建的。

## `AOP`

`AOP` 是 `Aspect Oriented Programming`，即面向切面编程。

它的价值主要在于：

1. 聚焦业务逻辑
2. 代码复用
3. 集中管理横切关注点
4. 在不破坏封装、不侵入业务代码的前提下增加额外能力

`AOP` 与 `OOP` 一样，本质上是一种编程范式。它并不强制规定具体实现方式。在 `.NET` 中，经常通过特性 `Attribute`、过滤器 `Filter` 等机制实现类似切面能力。

例如在 `ASP.NET Core` 中，可以通过异常过滤器集中处理全局异常，而不是在每个业务方法里重复写异常处理逻辑。

## 三层架构

三层架构强调“高内聚、低耦合”，通常把系统划分为：

1. 表示层（`UI`, `User Interface`）
2. 业务逻辑层（`BLL`, `Business Logic Layer`）
3. 数据访问层（`DAL`, `Data Access Layer`）

此外还会配合实体类库（`Model`）作为数据传递载体。

### 实体类库

实体类库通常用于承载系统中的数据模型，负责在层之间传递结构化数据。

## 中台

在现代软件系统里，中台通常与前台、后台相对应，指一组被多个业务系统复用的中间能力集合。

它的核心目标通常是：

1. 减少重复造轮子
2. 沉淀共性能力
3. 提升多个系统之间的复用效率
